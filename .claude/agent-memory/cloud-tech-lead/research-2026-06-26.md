## Edge cases worth memorizing

### Vercel
- **`max: "max"` is NOT a 1800s shortcut** — resolves to plan max (800s on Pro/Enterprise, 300s on Hobby). To opt into 30 min you must use the literal `1800` number, per-function.
- **Secure Compute does NOT support >800s during the beta** — if a tenant buys the Secure Compute add-on, we cannot promise 30-min functions.
- **HTTP/1.1 idle connections can timeout** during long responses (no HTTP/2 PING equivalent) — stream progress or heartbeat for long-running work.
- **For workloads >1800s, use Vercel Workflows** (pause/resume, persistent state, no duration limit on steps). Don't try to push Functions past 30 min.
- **Per-function >800s is REQUIRED during beta** — no project-level defaults above 800s.

### Turso
- **Parent schema DB pattern** (`turso db create --type schema` for parent, `--schema parent-db` for children) gives atomic migrations across all child DBs. The parent ALTER propagates to all children or none (atomic). Child DBs created with a schema CANNOT have individual DDL — all DDL must go through the parent.
- **Starter plan: 500 DBs free, Scaler $29/mo: 10,000 DBs.** Plenty for 30 tenants, scales to ~500.
- **ATTACH max 10 DBs per query.** Useful for cross-tenant queries (control plane analytics), not on the hot path.

### Upstash
- **`INCR` is atomic, NOT idempotent.** A retried call increments twice. The asymmetry with Stripe's `Idempotency-Key` is the load-bearing cause of drift in B4 §2.
- **Global DB writes cost × N regions.** 1 primary + 1 read = $0.4/100K writes. For 30 tenants on us-east-1, this is wasted money + eventual consistency risk on reads.
- **Global DB is eventual consistency (LWW).** Don't use it for billing reads — read the Stripe source of truth.

### Trigger.dev v4
- **Waitpoints** primitive gives native HTTP-callback idempotency keys + human-in-the-loop pause/resume. This is the missing piece for B4 §2 reconciliation: a Trigger job wrapped in a Waitpoint-Idempotency-Key will not double-execute on retry, even if the upstream HTTP request retries.
- **Warm starts 100-300ms** vs v3 "several seconds" cold — meaningful for sync UX paths.
- **v3 → v4 migration is minutes, not days.** README still says v3 — should say v4.

### Stripe
- **`Idempotency-Key` replay includes 500s.** Useful signal: if our retry logic gets a 500 back, we know Stripe already saw the request and we can safely retry to get the recorded state.
- **24h TTL on idempotency keys.** Choose keys with enough entropy to avoid collisions in a 24h window per tenant.
- **100 req/s per account default** (200 on request). If we send unbundled usage events for 30 tenants, we approach the limit. Bundle ("per 1000 transactions, send 1 record per 1000") is the documented workaround.
- **`timestamp` outside billing period = hard error** (except grace period for sum mode). Clock drift handling matters.

### Cloudflare DNS
- **Wildcards are first-label-only** — `*.example.com` ✓, `*.foo.example.com` ✓, `foo.*.example.com` ✗.
- **Multi-level by default** — `*.example.com` covers `abc.example.com` AND `123.abc.example.com`.
- **Specific records beat wildcards.**
- **Standard nameservers (our plan) are permissive** with empty-label descendants; **Advanced nameservers (Enterprise) follow strict RFC 4592.** If we ever upgrade to Enterprise, behavior shifts.
- **Wildcard certs require TXT DCV per hostname** when not using Delegated DCV.

### Cloudflare Workers
- **HTTP wall time UNLIMITED while client connected** — major change. Old 25s/300s envelope assumption is dead (limits doc updated 2026-06-25).
- **`waitUntil()` extends +30s after response/disconnect** — useful for fire-and-forget telemetry or audit logging.
- **Cron/Queue/DO alarms capped at 15 min wall.**
- **500 Workers per account on Paid plan** (1,000 on Enterprise) — well within 30 tenants + control plane.
- **1,000 routes per zone, 100 custom domains per zone** — wildcard route for beyond.
- **Worker startup time: 1 second.** Global scope must boot in 1s.
- **128 env vars per Worker on Paid, 5KB per var.** Fine for our secret bundles.
- **Log size: 256KB per request** — beyond that, context truncated.

### PASETO / PASERK
- **v4.public = EdDSA Ed25519.** Header `v4.public.`. No alg field — algorithm pinned by version.
- **v4.local uses BLAKE2b-derived subkeys** (Ek + Ak + counter nonce n2) with XChaCha20 + BLAKE2b-MAC. Not raw XChaCha20-Poly1305 — it's a PASETO-specific AEAD construction.
- **Algorithm lucidity assertion required** before use — must verify the key is intended for `v4.public` or `v4.local`. Prevents key-confusion attacks.
- **Footer optional** — base64url-encoded metadata after the payload.
- **PASERK k4.seal** wraps a 32-byte local PASERK key using X25519 + XChaCha20-Poly1305 (per the PKE spec). The control plane has the wrapping keypair, the tenant app has the recipient keypair.
- **⚠️ Tenant app must hold an X25519 private key** to `unseal` the bundle. Either the template bakes it in OR an out-of-band first exchange delivers it. NOT YET SPECIFIED in B3 — open question.

### Resend
- **Option A (single operator account + domain-scoped API keys)** matches our model. Risk: shared sender reputation.
- **Tags `tenant_id`** for webhook routing — natural pattern.
- **Subdomain warm-up schedule** is mandatory per Resend's guidance.
- **Eject = domain deletion + re-verification + DNS re-propagation window** — minutes to hours of mail interruption per tenant. Document this in C3's "4-6h DevOps" estimate.

## Gaps in the A/B/C taxonomy (load-bearing for v0)

These are gaps where the A/B/C docs don't yet cover a topic that's required by the spec:

- **A4 — Control plane runtime limits + Edge patterns.** Current A2 doesn't capture the new "wall-time unlimited while connected" model. Needs a dedicated doc on `POST /tenants` and `POST /exchange` streaming patterns.
- **B5 — Metering cost model.** Regional vs Global Upstash, bundle vs unbundle Stripe usage records, the unit economics of the metering pipeline at 30 → 100 tenants.
- **B6 — Waitpoint idempotency pattern.** Trigger.dev v4's Waitpoint primitive as the job-side idempotency layer, complementing Stripe's Idempotency-Key. Changes B4 §2 reconciliation story.
