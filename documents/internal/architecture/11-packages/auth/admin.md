# Admin plugin — global operator surface

The deep dive on the Better Auth `admin()` plugin. The one-paragraph summary is in [`plugins.md`](./plugins.md#admin--global-operator-surface-spec-219-220). Impersonation specifics live in [`impersonation.md`](./impersonation.md).

> **Scope reminder.** This plugin operates on the `user` table (GLOBAL — across every org, every session, every product surface). It does **not** touch `members` (org plugin) or any per-org data. Use it for the operator/admin panel. Use the org plugin for everything else.

## 1. What the admin plugin does (and doesn't)

### What it does

- **User management at the platform level** — list / search / create / update / remove users regardless of org membership.
- **Ban / unban** — block a user from signing in, with optional expiry. Revokes all of the user's existing sessions on ban.
- **Role management** — change `user.role` between the configured values (`admin`, `member` in our setup).
- **Session management per user** — list sessions, revoke one, revoke all.
- **Impersonation** — create a session that acts as the target user. See [`impersonation.md`](./impersonation.md).

### What it doesn't do

- **Org-scoped admin** — managing members of one org, transferring ownership, deleting an org. That's the org plugin (`member.role`).
- **Multi-tenant scoping** — the admin plugin has no notion of "region admin" or "tenant admin". Every operator sees every user.
- **Audit log** — Better Auth does not ship one. Our `databaseHooks` write to an `auditLog` table (M2+ for full coverage; M1 hooks wire only the existing events).
- **Bulk operations** — no `listUsers` "ban all" affordance. The UI must loop.
- **Custom RBAC statements for admin actions** — we use the built-in `user` and `session` statements. We don't define `project` or `billing` statements inside the admin plugin; those live in the org plugin.

## 2. Configuration in `auth.ts`

The plugin call (lines 222-226 of `packages/auth/src/auth.ts`) :

```ts
admin({
  defaultRole: 'member',
  adminRoles: ['admin'],
}),
```

Three load-bearing choices:

1. **`defaultRole: 'member'`** — every new user (via sign-up, admin `createUser`, or OAuth) gets `user.role = 'member'`. Alternative was `'user'` (Better Auth's built-in default); we picked `'member'` to align with org plugin terminology.
2. **`adminRoles: ['admin']`** — a user can call any `/admin/*` endpoint if `user.role === 'admin'`. We could expand this list (e.g. `['admin', 'superadmin']`) but currently don't.
3. **No `ac` / `roles` passed** — the plugin uses its built-in statements (`user`, `session`). We don't override these. Rationale in [`decisions/0002-admin-plugin-config.md`](./decisions/0002-admin-plugin-config.md).

### Schema additions (auto-generated)

The plugin adds 4 columns to the `users` table and 1 to the `sessions` table. Both are auto-generated by `pnpm auth:generate` and committed to `packages/database/src/schema/auth/index.ts`.

**`users`** (cf. lines 23-26 of the generated schema) :

| Column | Type | Purpose |
|---|---|---|
| `role` | `text` | The global role. Defaults to `'member'` per our config. |
| `banned` | `boolean` | `true` = blocked from sign-in. Default `false`. |
| `banReason` | `text` | Free-text reason. Displayed to the banned user via `bannedUserMessage` config. |
| `banExpires` | `timestamp` | Optional ban expiry. `null` = permanent. |

**`sessions`** (cf. line 46 of the generated schema) :

| Column | Type | Purpose |
|---|---|---|
| `impersonatedBy` | `text` | The admin user ID who started this impersonation session. `null` for normal sessions. |

## 3. Server-side endpoints — full list

All endpoints are mounted automatically by `auth.handler` at `/api/admin/*` (same Hono mount in `packages/api/src/hono/mount-auth.ts` as the rest of Better Auth). Every endpoint:

- Requires a valid session cookie (Better Auth enforces this).
- Requires `user.role` to be in `adminRoles` (Better Auth enforces this server-side).
- Accepts and returns JSON.
- Is callable from `apps/web` server components via `auth.api.<method>({ body, headers })` and from client components via `authClient.admin.<method>(...)`.

| Endpoint | HTTP | Body / Query shape | Returns |
|---|---|---|---|
| `listUsers` | GET | `query: { searchValue?, searchField?: 'email'\|'name', searchOperator?, limit?, offset?, sortBy?, sortDirection?, filterField?, filterValue?, filterOperator? }` | `{ users: User[], total: number, limit?: number, offset?: number }` |
| `getUser` | GET | `query: { id: string }` | `User \| null` |
| `createUser` | POST | `body: { email, password, name, role?, data?: Record<string, any> }` | `User` |
| `updateUser` | POST | `body: { userId, data: Record<string, any> }` | `User` |
| `setRole` | POST | `body: { userId, role: string \| string[] }` | `User` |
| `setUserPassword` | POST | `body: { userId, newPassword }` | `{ success: true }` |
| `banUser` | POST | `body: { userId, banReason?, banExpiresIn?: number /* seconds */ }` | `User` |
| `unbanUser` | POST | `body: { userId }` | `User` |
| `removeUser` | POST | `body: { userId }` | `{ success: true }` (hard delete from DB) |
| `listUserSessions` | GET | `query: { userId }` | `Session[]` |
| `revokeUserSession` | POST | `body: { sessionToken }` | `{ success: true }` |
| `revokeUserSessions` | POST | `body: { userId }` | `{ success: true }` |
| `impersonateUser` | POST | `body: { userId }` | `Session` (with `impersonatedBy` set + new cookie) |
| `stopImpersonating` | POST | (no body) | `{ success: true }` |
| `hasPermission` | GET | `body: { userId?, role?, permission?, permissions? }` | `{ success: boolean }` |

**Filter / search operators** (for `listUsers` `filterOperator` and `searchOperator`): `eq`, `ne`, `lt`, `lte`, `gt`, `gte`, `in`, `not_in`, `contains`, `starts_with`, `ends_with`.

**Pagination** (per the Better Auth docs) :

- Total pages: `Math.ceil(total / limit)`
- Current page: `(offset / limit) + 1`
- Next offset: `Math.min(offset + limit, total - 1)`
- Previous offset: `Math.max(0, offset - limit)`

## 4. Client-side wiring (`adminClient`)

`apps/web/src/lib/auth-client.ts` currently exports only the bare `createAuthClient`. To enable the admin namespace on the client:

```ts
// apps/web/src/lib/auth-client.ts
import { createAuthClient } from 'better-auth/react'
import { adminClient } from 'better-auth/client/plugins'

export const authClient = createAuthClient({
  baseURL: publicEnv.appUrl,
  plugins: [adminClient()],   // ← the only addition needed
})

// `authClient.admin.listUsers`, `authClient.admin.banUser`, etc.
// become available with full TypeScript types.
```

**No options needed** when the server uses built-in statements. If we later customize `ac` / `roles` server-side, we MUST pass the same to `adminClient({ ac, roles })` — otherwise type drift between server and client.

## 5. Server actions pattern in `apps/web`

Every admin mutation in the UI calls a **Next.js server action**, not an API route or `fetch` from the client. Rationale:

- Type-safe end-to-end (input is a typed function, not an HTTP shape).
- `revalidatePath` is in the same call as the mutation — no manual cache invalidation.
- `headers()` is available server-side without round-tripping the cookie.
- The re-verification is implicit (every server action re-reads the session).

### The pattern

```ts
// apps/web/src/app/(app)/admin/users/[userId]/actions.ts
'use server'

import { headers } from 'next/headers'
import { revalidatePath } from 'next/cache'
import { notFound } from 'next/navigation'
import { auth } from '@deessejs/auth'

async function requireAdmin() {
  const session = await auth.api.getSession({ headers: await headers() })
  if (!session || session.user.role !== 'admin') notFound()
  return session
}

async function assertNotSelf(adminId: string, targetId: string) {
  if (adminId === targetId) {
    throw new Error('FORBIDDEN_SELF_ACTION')
  }
}

export async function banUserAction(formData: FormData) {
  const session = await requireAdmin()
  const userId = String(formData.get('userId'))
  await assertNotSelf(session.user.id, userId)

  await auth.api.banUser({
    body: {
      userId,
      banReason: String(formData.get('reason') ?? 'No reason'),
      banExpiresIn: formData.get('expiresInSec')
        ? Number(formData.get('expiresInSec'))
        : undefined,
    },
    headers: await headers(),
  })

  revalidatePath('/admin/users')
  revalidatePath(`/admin/users/${userId}`)
}
```

### The role gate — `notFound()` not `redirect()`

The layout in `app/(app)/admin/layout.tsx` calls `requireAdmin` (same pattern) and on failure calls `notFound()` rather than `redirect()`. This is deliberate:

- `redirect("/dashboard")` for a non-admin trying to access `/admin` would confirm that `/admin` exists (oracle).
- `notFound()` returns the same 404 the user would get for `/admin/does-not-exist`, hiding the existence of the admin panel.

Same rationale as GitHub, Linear, Vercel — see [`03-web-app/admin.md`](../../03-web-app/admin.md) § "The role gate".

## 6. Interaction with the org plugin — the role ambiguity

This is the single most confusing part of the admin plugin for new contributors. The two plugins both have a notion of "role", but on **different fields**:

| | `user.role` (admin plugin) | `member.role` (org plugin) |
|---|---|---|
| **Where** | `users` table, single string column | `members` table, one row per (user × org) pair |
| **Scope** | GLOBAL — applies across every org | ORG-SCOPED — applies only within one org |
| **Values (our setup)** | `'admin'`, `'member'` | `'owner'`, `'admin'`, `'member'`, `'billing'` |
| **Who uses it** | Operator-level actions (ban, impersonate, see all users) | Per-org actions (manage members, transfer ownership, set billing) |
| **How to change** | `auth.api.setRole({ body: { userId, role: 'admin' } })` | `auth.api.organization.updateMemberRole({ body: { organizationId, memberId, role: 'admin' } })` |
| **Where it appears in UI** | `/admin/users/[userId]` (global role badge) | `/[orgSlug]/members` (per-org role) |

### The decision tree for "which role am I changing?"

```
Does the action affect the user across every org?
  → YES: it's user.role (admin plugin). Example: "Promote Alice to global admin".
  → NO: it's member.role (org plugin). Example: "Make Alice admin of Acme Corp".
```

### The naming collision

Both plugins use the string `'admin'`. They are **completely independent values** that happen to share a name. A user can have `user.role === 'member'` (not a global operator) AND `member.role === 'admin'` in org Acme (admin of that one org).

This is a UI hazard — the AdminSidebar in `/admin` and the MembersSidebar in `/[orgSlug]/admin` must label roles clearly:

- `/admin/users/[userId]` → "Global role" badge
- `/[orgSlug]/members` → "Acme Corp role" badge

Never both labelled "Role" — the labels are the only thing that distinguishes them for the user.

## 7. Options reference

Full reference for every option the `admin({...})` call accepts.

| Option | Type | Default | Our value | Effect |
|---|---|---|---|---|
| `defaultRole` | `string \| string[]` | `'user'` | `'member'` | Role(s) assigned to a newly-created user. Applies to sign-up, admin `createUser`, OAuth (if `defaultRole` is honored). |
| `adminRoles` | `string[]` | `['admin']` | `['admin']` | List of `user.role` values that grant admin access to all `/admin/*` endpoints. |
| `adminUserIds` | `string[]` | `[]` | `[]` (not set) | Whitelist of user IDs that are always admin, regardless of `user.role`. Useful for bootstrapping. |
| `impersonationSessionDuration` | `number` (seconds) | `3600` (1h) | default | Duration of an impersonation session. Override only if the buyer needs longer debug sessions. |
| `defaultBanReason` | `string` | `'No reason'` | default | `banReason` set on `banUser` if not provided. |
| `defaultBanExpiresIn` | `number \| undefined` | `undefined` (permanent) | default | `banExpiresIn` set on `banUser` if not provided. |
| `bannedUserMessage` | `string` | `'You have been banned...'` | default | Message shown to a banned user trying to sign in. |
| `customSyntheticUser` | `function` | (none) | not set | Hook for `requireEmailVerification` / `autoSignIn: false` synthetic user. **Required if you turn those on later — see Common gotchas §3.** |

The `ac` and `roles` options are **not listed** — they only exist when you define a custom access-control matrix. We don't, so we don't pass them.

## 8. Common gotchas

| # | Gotcha | Severity | Mitigation |
|---|---|---|---|
| 1 | Confusing `user.role` with `member.role` in code | HIGH | Always label clearly in UI. Code comment near every `setRole` call. See §6. |
| 2 | Self-action lock-out (admin removes own role → no one left) | HIGH | Server-side guard `assertNotSelf` on `setRole`, `banUser`, `removeUser`, `revokeUserSessions`. See [`03-web-app/admin.md`](../../03-web-app/admin.md). |
| 3 | `requireEmailVerification: true` + `customSyntheticUser` missing | MEDIUM | If we ever turn on `autoSignIn: false` after signup, set `customSyntheticUser` to include `role`, `banned`, `banReason`, `banExpires` — otherwise fake signups crash. We don't currently need this. |
| 4 | `impersonateUser` for an admin target silently no-ops (PR #6454) | MEDIUM | Documented default behavior. To override, grant `impersonate-admins` permission via custom RBAC. |
| 5 | `removeUser` is a hard delete — sessions and accounts are CASCADE-deleted by Drizzle | MEDIUM | UI MUST confirm with a typed-email dialog before this call. |
| 6 | `adminClient` not in `auth-client.ts` → `authClient.admin` is `undefined` | HIGH | The single Edit to `auth-client.ts` adding `plugins: [adminClient()]` is required before any UI work. |
| 7 | Cookie refresh via `asResponse: true` workaround does NOT propagate through `impersonateUser` automatically | LOW | After `impersonateUser`, the response carries a new `set-cookie`. The Hono mount in `packages/api/src/hono/auth-context.ts` propagates these via the existing middleware loop. Verify in tests. |
| 8 | `adminRoles: ['admin']` is matched literally — case-sensitive | LOW | Don't rename the role to `'Admin'` without updating `adminRoles` to match. |
